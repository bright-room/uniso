# /plan スキル 使い方ガイド

## 概要

GitHub Issue を参照して実装プランを作成し、Issue コメントとして投稿するスキル。

## 使い方

```
/plan 42        # Issue #42 の実装プランを作成し、Issue コメントに投稿
/plan 123       # Issue #123 の実装プランを作成し、Issue コメントに投稿
```

Issue 番号の指定が必須。

## 出力先

**Issue コメント**として投稿する（`<!-- claude:plan -->` マーカー付き）。

API エラー時のみ `.claude/outputs/plans/PLAN-<Issue番号>-<タイトル>.md` にフォールバック出力する。

## プランの更新

同じ Issue 番号で再度 `/plan` を実行すると、既存のプランコメントを検出して上書き更新する。

- 最新のプランが常にコメントの本文に表示される
- 変更履歴は折りたたみ（`<details>`）で末尾に残る

```
/plan 42        # 初回: プランコメントを新規投稿
# → Issue 上でフィードバックを受ける
/plan 42        # 2回目: 既存コメントを更新（変更履歴付き）
```

## 処理の流れ

1. **Issue の取得** — `gh issue view` で Issue の要件を取得
2. **既存プランの確認** — Issue コメントから `<!-- claude:plan -->` マーカーを検索
3. **コードベースの調査** — 関連するコード、既存パターン、影響範囲、ドメインマネージャーとの関連を調査
4. **実装プランの作成** — 具体的な実装ステップ、テスト戦略を設計
5. **プランの投稿** — Issue コメントとして投稿（既存プランがあれば更新）

## 実装プランの構成

1. **概要** — 要件の要約と実装方針
2. **設計判断** — 複数アプローチのトレードオフ分析（必要な場合のみ）
3. **影響範囲** — パッケージ別の影響マトリクス
4. **ファイル構成** — 変更・追加されるファイルのツリー表示
5. **実装ステップ** — Phase（論理的なまとまり）→ Step（具体的な作業単位）の階層構造
6. **ドキュメント更新** — 実装に伴い更新が必要なドキュメント（README, CLAUDE.md 等）
7. **テスト戦略** — テスト種別ごとの方針（Vitest / Playwright / Storybook）
8. **確認事項** — Issue オーナーへの確認事項（必要な場合のみ）
9. **今後の展望** — スコープ外の関連拡張（必要な場合のみ）
