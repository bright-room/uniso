# Implementation Plan Skill

GitHub Issue を参照し、コードベースを調査した上で実装プランを作成する。プランは Issue 上のコメントとして投稿・管理する。

## 前提条件

- `gh` CLI が認証済みであること

## 手順

### 1. Issue の取得

引数に Issue 番号を指定する（必須）。

`gh issue view <issue-number>` で Issue のタイトル、本文、ラベル、コメントを取得し、要件を正確に理解する。

### 2. 既存プランコメントの確認

Issue のコメント一覧を取得し、既にプランコメントが投稿されているかを確認する。

```bash
gh api repos/{owner}/{repo}/issues/<issue-number>/comments --jq '.[] | select(.body | contains("<!-- claude:plan -->"))'
```

- `<!-- claude:plan -->` マーカーを含むコメントが見つかった場合、そのコメント ID を記録する（後で更新に使う）
- 既存プランがある場合は、その内容も確認し、何が変更されるかを把握する

### 3. コードベースの調査

Issue の要件に基づいて、関連するコードを徹底的に調査する。

- **既存の実装パターン**: 同様の機能がどのように実装されているかを確認する
- **影響範囲**: 変更が影響するパッケージ（`apps/desktop`、`packages/shared`、`packages/ui`）を特定する
- **拡張ポイント**: 以下の既存クラス・パターンとの関連を確認する
  - ドメインマネージャー: AccountManager, SessionManager, LinkRouter, I18nManager, IdentityManager, ServicePluginRegistry
  - データ層: AccountRepository, SessionRepository, SettingsRepository, ServicePluginRepository
  - IPC チャネル（`apps/desktop/src/main/ipc-handlers.ts`）
  - React コンポーネント（`packages/ui/src/features/`, `packages/ui/src/primitives/`）
  - 新しい SNS サービス追加の場合: ServicePluginRegistry のメタデータ拡張パターン
- **テスト構成**: 既存のテスト構成（Vitest ユニットテスト、Playwright E2E テスト、Storybook）を確認する

### 4. 実装プランの作成

以下の観点で実装プランを設計する:

- **実現可能性**: 要件が現在のアーキテクチャで実現可能かを検証する
- **設計判断**: 複数のアプローチがある場合は、各アプローチのトレードオフを分析する
- **段階的な実装**: 変更を論理的なステップに分割し、各ステップの依存関係を明確にする
- **テスト戦略**: 各変更に対するテスト方針を策定する

### 5. 実装プランの投稿

実装プランを Issue コメントとして投稿する。

#### 新規投稿（既存プランコメントがない場合）

```bash
gh api repos/{owner}/{repo}/issues/<issue-number>/comments \
  --method POST \
  --field body='<プラン本文>'
```

#### 更新（既存プランコメントがある場合）

既存のプランコメントを編集し、最新版に更新する。変更履歴は折りたたみで残す。

```bash
gh api repos/{owner}/{repo}/issues/comments/<comment-id> \
  --method PATCH \
  --field body='<更新されたプラン本文>'
```

更新時は、プラン本文の末尾に以下の変更履歴セクションを追加・更新する:

```markdown
<details><summary>変更履歴</summary>

- v2: ○○を修正（指摘: ××が不足）
- v1: 初版作成
</details>
```

#### API エラー時のフォールバック

API 呼び出しが失敗した場合は、ローカルファイルにフォールバック出力する。

- 出力先: `.claude/outputs/plans/PLAN-<Issue番号>-<Issueタイトルをケバブケースに変換>.md`
- ユーザーに API エラーが発生した旨と、手動で Issue にコメントをコピーするよう案内する

## 注意事項

- Issue の要件を正確に理解し、過不足のないプランを作成すること
- 推測ではなく、実際にコードを読んで確認した事実に基づいてプランを作成すること
- 実装ステップは具体的に記述し、各ステップで変更するファイルパスを明示すること
- 既存のアーキテクチャパターンや命名規則に従った設計にすること
- モノレポの各パッケージの責務を理解し、適切なパッケージにコードを配置すること
  - `packages/shared`: フレームワーク非依存のビジネスロジック（Electron/React に依存しない）
  - `packages/ui`: 純粋な React コンポーネント（Electron API に依存しない）
  - `apps/desktop`: Electron 固有のコード（メインプロセス、IPC、プリロード）
- 不明点や判断が必要な事項は「確認事項」セクションに明記すること
- 変更のない「設計判断」「確認事項」「今後の展望」セクションは省略してよい
- 「今後の展望」は Issue のスコープ外の関連拡張を記載する。実装プラン策定時にコードベースを調査する中で見つけた改善機会や、自然な次のステップとなる機能拡張を挙げること
- 実装に伴うドキュメント（`README.md`、`CLAUDE.md`）の更新を必ず「ドキュメント更新」セクションに記載すること。新機能・設定変更・アーキテクチャ変更がある場合、対応するドキュメントの更新箇所を具体的に特定すること
